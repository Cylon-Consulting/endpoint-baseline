# endpoint-baseline

tracking against upstream WinUtil. Public by design: it contains no
credentials, hostnames, or client data â€” integrity is enforced by SHA256
pinning in `manifest.json`, and the fleet launcher refuses to execute
anything whose hash does not match.

Two independent jobs: a **drift checker** (awareness, GitHub Action, weekly) and a
**fleet launcher** (enforcement, ConnectWise RMM, monthly after Patch Tuesday).
The checker never touches endpoints; the launcher never reads upstream directly.

## The check model (three layers + awareness)

| Layer | Question | Source compared |
|---|---|---|
| Membership | Did Standard's tweak ID list change? | upstream `preset.json` vs `approvedPreset` |
| Definition | Did an approved tweak's body change (registry/services/scripts)? | per-tweak SHA256 of canonicalized `tweaks.json` entry vs `tweakHashes` |
| Existence | Was an approved tweak deleted upstream? | approved ID missing from `tweaks.json` |
| FYI | New Essential Tweaks we never reviewed | category scan, never triggers alert |

Hashes are computed over a canonical serialization (sorted keys, fixed
escaping) done by the script itself, so they are stable across PS 5.1/7 and
across baseline/check runs. `tweaks.json` contains literal newlines inside
JSON strings; the script escapes control chars inside string literals before
parsing.

**Release vs main:** `christitus.com/win` serves the latest *release* build,
which lags the `main` branch. The checker therefore defaults to the latest
release tag (`-Branch latest-release`, auto-resolved via the GitHub API) so
it tracks what machines actually receive. Example of the gap, observed
2026-06-10: release 26.05.12 still ships `WPFTweaksPowershell7Tele` in
Standard, while main has removed it and added
`WPFTweaksDeliveryOptimization` â€” those changes will surface as drift only
when the next release is cut.

## Files

- `Check-WinutilDrift.ps1` â€” weekly. `-UpdateBaseline` to seed/re-approve.
  Exit 0 clean / 1 drift / 2 error. Report at
  `%ProgramData%\WinutilDrift\last-report.txt` and stdout.
- `baseline.json` â€” generated by `-UpdateBaseline`. To adopt an upstream
  preset addition: add the ID to `approvedPreset` by hand, run
  `-UpdateBaseline` to capture its hash.
- `Invoke-EndpointBaseline.ps1` â€” the only thing ConnectWise holds. Static.
  Takes `-ManifestUrl` (+ `-AuthToken` from an RMM credential variable).
  Exit 0 ok / 2 integrity or download failure (nothing executed) / 3 payload
  failed. Transcript logs in `%ProgramData%\EndpointBaseline\logs`.
- `manifest.sample.json` â€” template; `manifest.json` is the live control
  file. Pins payload URL **by commit SHA** + SHA256, config URL + SHA256,
  and the argument template. Changing fleet behavior = commit a new
  manifest. Never edit the RMM script.
- `payload/Invoke-EfficiencyBaseline.ps1` â€” the fleet payload: telemetry/
  privacy reduction + low-risk efficiency tweaks, config-driven,
  self-contained (no WinUtil at runtime). Full change inventory and
  rationale in `payload/SETTINGS.md`.
- `payload/efficiency-config.json` â€” all payload toggles + appx removal
  list. Data-only fleet changes happen here.
- `.github/workflows/drift-check.yml` â€” weekly drift check (Wednesdays
  07:00 UTC). Opens/updates a `winutil-drift` issue on drift; commits a
  keepalive stamp each run so GitHub's 60-day inactivity rule never
  disables the schedule. Manual dispatch with `update_baseline=true`
  re-approves current upstream as the new baseline.

## Why hash verification in the launcher

The RMM script trusts only the manifest URL (in your access-controlled repo).
The manifest pins exact SHA256s of everything else, so even if the payload
host or a redirect is compromised, a tampered file fails `Assert-FileHash`
and exits 2 without executing. Updating "what runs" is a reviewed commit in
your repo, with git history as the audit trail.

## Cadence

- Drift check: weekly, Wednesday after Patch Tuesday bias. Alert-only.
- Fleet apply: monthly, 2-3 days after Patch Tuesday, off-hours window,
  plus once at device onboarding.
